# Skill: research-outreach

> **Architecture:** One of 7 skills in the Sales Agent. See `README.md` for the overview.
> **Shared rules:** `CLAUDE.md` (greeting, tone, signatures).
> **Configuration:** Your research approach is defined in `knowledge/research-config.md`.
> **Related skill:** `lead-recovery.md` provides the lead list, this skill executes the outreach.

---

## Purpose

Curated high-quality outreach: for a **manually selected** lead list, research each lead's website (or business), generate a structured report based on your configured research type, embed the top findings as an HTML table in a personalized email, and save as a Gmail draft. Full reports land in `output/research-reports/` as markdown.

**Difference from `follow-up-loop`:** Not mass sending. Deliberately curated. 5-25 leads per run. More effort per lead, significantly higher reply rate.

**Industry-agnostic:** You define what to research in `knowledge/research-config.md`. Default examples include SEO audits, UX reviews, brand positioning analyses, tech stack audits, content strategy reviews, competitive analyses, and more. Pick one (or define your own) that aligns with what you sell.

## Trigger
Manual with a lead list. Invocation examples in `prompts/invoke-skill.md`.

## Output
- `output/research-reports/<domain>.md` — full research report per lead
- Gmail HTML drafts with embedded top-3 findings
- `tracker` rows with `status=drafted` and `RES:` prefix in `notes_summary`

## Stopping
When all leads in the input list are processed. On errors (offline domains, expired SSL, 404): silent skip with log.

---

## Setup

0. **Load learnings** — Read `knowledge/learnings.md`. Section A informs greeting/tone, the most recent ~20 entries of Section B flag audit patterns you've seen before (recurring findings, what hooks landed), Section C lists distilled rules. Universal requirement — see `program.md`.

1. **Receive lead list** (from the user or from `lead-recovery` output):
   - Format: `[{email, firstname, lastname, company, domain, lead_status, hubspot_id?}, ...]`

2. **Tracker check:** `npx tsx src/tracker.ts exists <email>` for each lead. If yes → decide whether research-outreach should still run (it's a **different** campaign than follow-up — a second outreach with a concrete hook is acceptable if the first was generic and got no reply).

3. **Load HubSpot notes** (optional, for context in the email body).

4. **Read `knowledge/research-config.md`** to know WHAT to audit and HOW to frame findings.

## Per-Lead Workflow

### Step 1 — Validate domain
- If `domain` in the input is empty: derive from the email address (`info@example.com` → `example.com`)
- If the domain is obviously invalid (free mailers like gmail.com, outlook.com): SKIP, log as `no_domain`

### Step 2 — Research the target
Use the research approach defined in `knowledge/research-config.md`. The audit can be any of:
- **SEO audit** — technical SEO, content, metadata, page structure
- **UX review** — design, conversion flow, mobile experience, accessibility
- **Brand positioning** — messaging clarity, value proposition, differentiation
- **Tech stack audit** — frameworks, performance, security, third-party tools
- **Content strategy** — content depth, blog frequency, topical authority
- **Competitive analysis** — how they compare to competitors in their space
- **Custom** — whatever you define in `research-config.md`

**Execution:**
- **MCP:** `WebFetch` on `https://<domain>` with your configured audit prompt (any MCP-capable harness)
- **CLI:** `npx tsx src/tools/webfetch.ts audit --url https://<domain> --type <your-type>`

**Error handling:**
- 404 / connection refused / SSL expired / bounce → SKIP, log as `domain_unavailable`
- 301 redirect to another domain → follow the redirect, audit the new domain
- Audit successful but empty/uninformative → SKIP, log as `audit_empty`

### Step 3 — Select top-3 findings
From the audit output, pick the 3 most impactful issues (by impact). Per finding: **1 sentence description + 1 sentence impact** (what it costs them).

### Step 4 — Save full report
Write markdown file to `output/research-reports/<domain-slug>.md`:

```markdown
# Research Report: <domain>
**<Company>** — <Industry / description>
Created: <ISO date>

## Overall Assessment
[Table with area + rating + priority per audit point]

## What's Working
[Bullet list of strengths — important for trust]

## Critical Issues
[Top 3 with detailed description + suggested solution]

## Additional Optimization Points
[Medium priority findings]

## Quick Wins
[What's immediately actionable]

---
*Research report generated by YOUR_NAME (YOUR_DOMAIN)*
```

### Step 5 — Choose email template

**Template A — Formal, critical issues found:**
- **When:** Lead status `ATTEMPTED_TO_CONTACT`, `NEW`, `UNQUALIFIED`, or unclear/conservative industry
- **Subject pattern:** `[N] findings on [domain] that may cost [target audience]`
- **Tone:** "I took a look at your website / business and noticed..."

**Template B — Casual, business is solid but has potential:**
- **When:** Lead status `CONNECTED` AND not a conservative industry AND young tech founder context
- **Subject pattern:** `[Company]: solid foundation, but [X]% potential on the table`
- **Tone:** "Hey [first name], I took another look at [domain]. Things look technically better than most in your space..."

**Template C — Casual, documented prior contact:**
- **When:** Lead status `CONNECTED` AND not a conservative industry AND we had documented casual contact before
- **Subject pattern:** `[First name], your site is leaving [X] on the table — [N] findings`
- **Tone:** "Hey [first name], we talked in [month] about your project. I took another look at [domain]..."

**Greeting override:** For conservative industries (tax, law, medicine, finance), ALWAYS use Template A even if lead status is CONNECTED. See `CLAUDE.md` and `knowledge/learnings.md`.

### Step 6 — Build HTML email body

**Standard structure:**
```html
<div style="font-family: Arial, sans-serif; font-size: 14px; color: #333; line-height: 1.6;">
<p>[Greeting + first name],</p>
<p>[Hook — 1-2 sentences of context or introduction]</p>

<table style="border-collapse: collapse; width: 100%; margin: 16px 0;">
  <tr style="background: #f8f8f8;">
    <td style="padding: 10px; border: 1px solid #ddd;"><strong>1. [Issue]</strong></td>
    <td style="padding: 10px; border: 1px solid #ddd;">[Description + impact]</td>
  </tr>
  <tr>
    <td style="padding: 10px; border: 1px solid #ddd;"><strong>2. [Issue]</strong></td>
    <td style="padding: 10px; border: 1px solid #ddd;">[Description + impact]</td>
  </tr>
  <tr style="background: #f8f8f8;">
    <td style="padding: 10px; border: 1px solid #ddd;"><strong>3. [Issue]</strong></td>
    <td style="padding: 10px; border: 1px solid #ddd;">[Description + impact]</td>
  </tr>
</table>

<p>[What's working well — 1-2 positive points for trust]</p>
<p>[Soft CTA: "Shall I send you the full analysis?" or "Do you have 15 minutes?"]</p>

<p style="margin-top: 24px;">
[Kind regards / Best]<br>
YOUR_NAME<br>
<a href="https://YOUR_DOMAIN" style="color: #333;">YOUR_DOMAIN</a>
</p>
</div>
```

### Step 7 — Create Gmail draft
- **MCP:** `mcp__claude_ai_Gmail__gmail_create_draft` with `contentType=text/html` ← **IMPORTANT**, different from follow-up-loop
- **CLI:** `npx tsx src/tools/gmail.ts draft create --to <email> --subject "..." --body "<html...>" --content-type text/html`

### Step 8 — Tracker update
```bash
npx tsx src/tracker.ts append "<email>\t<firstname>\t<lastname>\t<company>\t<lead_status>\tRES: <domain> - <top1-finding>\t<draft_id>\tdrafted\t<ISO timestamp>"
```

**Marker:** `notes_summary` starts with `RES:` so you can later distinguish whether a lead was drafted via `follow-up-loop` or `research-outreach`.

```bash

  --text "Audited N domains, drafted M, skipped K. Audit type: <type>. Top finding: <most common>"
```

**Observation (if ≥3 audits surfaced the same finding, a segment showed an unusual audit pattern, or a hook/template worked differently than Section A suggests):**
```bash
npx tsx src/learnings.ts append observation --skill research-outreach \
  --headline "<short pattern name>" \
  --context "<batch description: audit type, industries, count>" \
  --observed "<quantitative finding pattern>" \
  --apply "<concrete rule for next run>"
```

Write observation **instead of** the heartbeat. See `program.md` for the universal teardown rule.

## Constraints

### Allowed
- Read HubSpot notes (for context)
- Fetch external websites via WebFetch / CLI
- Write markdown reports to `output/research-reports/`
- Create HTML drafts in Gmail
- Update `tracker`

### Forbidden
- Send emails
- "Invent" audit findings if the domain is unreachable — skip instead
- Use wrong personalized details from stale notes
- More than top-3 findings in the email body — the rest belongs in the report markdown
